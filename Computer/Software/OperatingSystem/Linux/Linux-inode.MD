Linux inode
-------------------
inode (index node) 索引节点。

操作系统的文件数据除了实际内容（数据）之外，通常还有其他的拓展属性（元信息）。
比如：Linux操作系统的文件权限与文件属性。
文件系统通常将文件的`实际内容`和`拓展属性`分别存放在`block`和`inode`中。

### `block`和`inode`

文件的实际数据存储在`block`中，文件的元信息（比如：文件的创建者/创建日期/大小/权限等）存储在`inode`中。

#### `block`
文件存储在硬盘上，硬盘的最小存储单位叫`sector`（扇区），每个`sector`存储512字节。
操作系统读取硬盘文件时，不会一个个`sector`去读，而是一次性连续读取多个`sector`，
一般连续8个`sector`构成一个`block`（块），即一个块4KB。
`block`是文件存取的最小单位。

#### `inode`
`inode`包含文件的元信息（不含文件名），具体来说有以下内容：
　　* 文件的字节数
　　* 文件拥有者的User ID
　　* 文件的Group ID
　　* 文件的读、写、执行权限
　　* 文件的时间戳，共有三个：ctime指inode上一次变动的时间，mtime指文件内容上一次变动的时间，atime指文件上一次打开的时间。
　　* 链接数，即有多少文件名指向这个inode
　　* 文件数据block的位置

##### inode号码
表面上，用户通过文件名打开文件，实际上，系统内部将这个过程分为三步：
1. 根据文件名，获取inode号码；
2. 根据inode号码，获取inode信息；
3. 根据inode信息，验证权限并找到文件数据所在的block，读出数据。

示例：查看文件xxx.txt的inode号码
`ls -i xxx.txt`
`stat xxx.txt |grep Inode`

##### inode大小与总数
inode本质也是一个文件，inode存储也会消耗硬盘空间。
格式化时，操作系统会自动将硬盘分为两个区域：
 - data区  存放data实际数据
 - inode区  存放inode元信息

一般而言，inode不会太大，通常是128字节或256字节。
inode总数在格式化分区时就确定了（根据数据块的数量确定）。

所以一般不太关注inode所占磁盘空间的大小，重点关注inode总数。

示例：查看硬盘分区的inode总数和已使用情况
`df -i`

** inode耗尽 **
由于硬盘分区的inode总数在格式化后就已经固定，而每个文件必须有一个inode，因此就有可能发生inode用光，但硬盘空间还剩不少，却无法创建新文件。

修复：找出那些大量占用inode的垃圾文件，删除之。

### 硬链接 与 软链接
每个文件名都有一个inode号码。
Unix/Linux系统允许多个文件名指向同一个inode号码。

#### 硬链接 
`ln source.txt target.txt `
原文件source.txt与目标文件target.txt的inode号码相同，即指向同一个inode。

inode信息中有一项叫做"链接数"，记录指向该inode的文件名总数，这时链接数就会加1。
反过来，删除一个文件名，就会使得inode中的"链接数"减1。
当这个值减到0，表明没有文件名指向这个inode，系统就会回收这个inode号码，以及其所对应block区域。

删除source.txt，链接数不为0，因为target.txt还指向着inode，访问target.txt仍能读取内容。
写入source.txt会影响target.txt的读取，反之亦然。

这里顺便说一下目录文件的"链接数"。
创建目录时，默认会生成两个目录项："."和".."。
前者(.)的inode号码就是当前目录的inode号码，等同于当前目录的"硬链接"；
后者(..)的inode号码就是当前目录的父目录的inode号码，等同于父目录的"硬链接"。所以，任何一个目录的"硬链接"总数，总是等于2加上它的子目录总数（含隐藏目录）。

#### 软链接 （快捷方式）
`ln -s source.txt target-s.txt `
创建软链接时，target-s.txt指向的不是source.txt的inode，而是source.txt的文件名。

删除target-s.txt不会影响source.txt。
删除source.txt，target-s.txt无法正常访问。

### inode的特殊作用

由于inode号码与文件名分离，这种机制导致了一些Unix/Linux系统特有的现象：
　　1. 有时，文件名包含特殊字符，无法正常删除。这时，直接删除inode节点，就能起到删除文件的作用。
　　2. 移动文件或重命名文件，只是改变文件名，不影响inode号码。
　　3. 打开一个文件以后，系统就以inode号码来识别这个文件，不再考虑文件名。因此，通常来说，系统无法从inode号码得知文件名。

第3点使得软件更新变得简单，可以在不关闭软件的情况下进行更新，不需要重启。
因为系统通过inode号码，识别运行中的文件，不通过文件名。
更新的时候，新版文件以同样的文件名，生成一个新的inode，不会影响到运行中的文件。
等到下一次运行这个软件的时候，文件名就自动指向新版文件，旧版文件的inode则被回收。
